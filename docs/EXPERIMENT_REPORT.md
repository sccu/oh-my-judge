# Anti-Bot Bypass Experiment Report (2026-03-11)

## 1. 실험 개요
본 실험은 강력한 봇 탐지 솔루션(Akamai Premier 등)이 적용된 국내 주요 이커머스 서비스(쿠팡, 네이버 쇼핑)를 대상으로, LLM 에이전트가 사람의 행동을 안정적으로 재현할 수 있는지 검증하기 위해 수행됨.

## 2. 주요 시행착오 및 결과

### 2.1. Scrapling (StealthyFetcher) 시도
- **방법**: 최신 TLS/JA4 스푸핑 라이브러리인 Scrapling을 사용하여 접속.
- **결과**: **부분 실패 (Soft Block)**.
- **분석**: 초기 TLS 핸드쉐이크는 통과하여 200 OK를 받았으나, 브라우저 런타임에서 실행되는 Akamai의 'Sensor Data' 수집 스크립트가 자동화 흔적을 감지함. 결과적으로 상품 리스트 대신 `sec-if-cpt-container` (챌린지 컨테이너)가 포함된 페이지가 로드됨.

### 2.2. Playwright + Camoufox (Firefox-based) 시도
- **방법**: C++ 레벨에서 핑거프린팅을 변조하는 Camoufox 브라우저 사용.
- **결과**: **실패**.
- **분석**: Firefox 기반 우회는 Chromium 기반보다 탐지율이 낮을 것으로 기대했으나, 국내 서비스 특유의 Chromium 최적화 보안 로직에 걸려 챌린지 페이지로 리다이렉트됨.

### 2.3. OpenChrome (Remote Debugging) + CDP 시도
- **방법**: 실제 유저가 사용하는 일반 크롬을 9222 포트로 띄우고 Playwright로 연결.
- **결과**: **조건부 성공 후 실패**.
- **분석**: 
    - **쿠팡**: 메인 페이지 진입은 성공했으나, 검색어 입력 후 검색 결과 페이지로 넘어가는 과정에서 '행동 분석(Behavioral Analysis)'에 의해 차단됨. 단순한 타이핑/클릭 지연만으로는 부족함을 확인.
    - **네이버 쇼핑**: `domcontentloaded`는 성공했으나, 실제 상품 리스트 렌더링 직전에 차단 화면 노출. 네이버의 자체 탐지 로직이 CDP 연결 흔적이나 환경적 요인(Headless 여부 등)을 정밀하게 체크함.

## 3. 기술적 한계 및 발견 사항
1. **Headless의 한계**: CLI 환경(`headless=True`)에서의 실행은 실제 유저의 GPU 가속, 캔버스 렌더링 특성 등과 미세한 차이를 보이며, 이는 고도화된 봇 매니저의 주요 먹잇감이 됨.
2. **IP Reputation**: 데이터센터 IP 사용 시, 아무리 브라우저 지문을 완벽하게 변조해도 IP 대역 자체의 신뢰도 점수가 낮아 챌린지 발생 확률이 급격히 높아짐.
3. **인터랙션의 정교함**: 단순한 랜덤 딜레이를 넘어서는, 실제 인간의 불규칙한 마우스 가속도 및 비선형 궤적이 필수적임.

## 4. 향후 해결 전략 (Next Steps)
1. **Residential/Mobile Proxies 도입**: IP 신뢰도 문제를 해결하기 위해 4G/5G 모바일 프록시 적용 필수.
2. **GUI 기반 로컬 재현**: CLI/서버 환경이 아닌, 실제 데스크탑 환경에서 `headless=False` 모드로 실행하여 브라우저 환경의 완벽한 일관성 확보.
3. **Nodriver 검토**: CDP를 통한 연결 방식 중에서도 가장 탐지율이 낮은 `nodriver` 라이브러리를 Action Layer 엔진으로 검토.
4. **유료 우회 서비스 활용**: 기업용 스텔스 브라우저 서비스(Browserless, Bright Data) 연동을 통한 인프라 수준의 해결.

---
**결론**: 현재의 오픈소스 기술 조합만으로는 Akamai Premier 수준의 실시간 행동 분석을 100% 우회하기 어려우며, 향후 **IP Reputation 개선**과 **완전한 유저 환경(GUI)**에서의 실행이 병행되어야 함.
